## Scope
F1 is the large language-and-stdlib completion phase that moves PulseCode from its current partial Java-style surface to a substantially Java-close language ecosystem. By F1 exit, the goal is not just "more library classes" but a usable, operational Java-like syntax and runtime model across lexer, parser, semantics, IR/backend/runtime lowering, standard library APIs, and the validation/CI guardrails that prove those features are real enough to support the later self-sustained-hosting transition.

This board is grounded in the current repo state:
- many foundational stdlib classes already exist in `stdlib/src`
- significant Java-like OOP support already exists (`extends`, `implements`, `interface`, `abstract`, `final`, `super`, `@Override`, `instanceof`)
- native execution already exists
- but the language surface is still incomplete relative to Java, and many stdlib APIs remain shallow

## F1 End-State Intent
At F1 closure, PulseCode should have:
- a broadly operational Java-like syntax surface for mainstream application code
- enough frontend + backend support that those syntax features really build and run, not just parse
- a materially expanded standard library that feels recognizably Java-like
- integrated conformance/fixture/backend/package validation for the shipped F1 surface
- explicit documentation of what remains intentionally non-Java or deferred past F1
- a cleaner separation between public user-facing `stdlib` growth and the later `authorlib` elevation work needed for compiler/runtime self-sustainment

## Current Baseline (Observed)
Already present in `stdlib/src`:
- `pulse.lang`: core object/string/wrapper/exception/console classes and interfaces
- `pulse.collections`: `Array`, `Collection`, `List`, `ArrayList`, `LinkedList`, `Set`, `HashSet`, `Map`, `HashMap`, `Queue`, `Deque`
- `pulse.io`: `File`, `Files`, `InputStream`, `OutputStream`, `Path`
- `pulse.math`: `Math`, `Random`
- `pulse.time`: `Instant`, `Duration` (with broader date/time and formatting families still to be expanded)
- `pulse.rt`: `Intrinsics`

Already present in the language/compiler at a meaningful baseline:
- packages/imports/static imports
- classes, interfaces, inheritance, abstract/final, override validation
- fields, methods, constructors
- control flow: `if`, `while`, `do-while`, `for`, `switch`, `break`, `continue`, `return`
- member access/calls, static-vs-instance rules, constructors, `super`, `this`
- boxing/unboxing baseline, wrappers, explicit numeric casts, `instanceof`
- IR lowering, native backend, runtime-backed executable builds

Known major gaps relative to a Java-close F1 target:
- several Java syntax families still missing or partial at token/parser/semantic/runtime level
- shallow stdlib APIs even where classes already exist
- collections remain non-generic and specialized around current runtime shapes
- exception classes exist, but language-level exception flow is not complete
- Java-close literals/tokens/operator surface is incomplete
- several core language declarations and modifiers are still unsupported or only reserved
- Java-adjacent utility families such as `Scanner`, `Objects`, concurrency baselines, and networking baselines are not yet explicitly tracked enough
- full Java-style reflection/invocation is now explicitly pulled into `F1-97`
  before the compiler/runtime rewrite and should not be implied by the current
  `Class` baseline until that work is actually complete
- desktop UI families (`awt` / `swing`) are not part of F1 scope
- generics scope, exception runtime strategy, and some utility/network/concurrency sub-policies still need explicit F1 boundaries
- conformance/docs currently do not define a full Java-close supported subset

## F1 Strict Done Policy
- F1 starts with a current-state audit of what already exists and whether it is genuinely runtime/backend implemented.
- F1 is not complete when syntax merely parses; syntax counts only when semantics, lowering, runtime behavior, fixture evidence, and task-local docs exist where relevant.
- `Done (Locked)` means tokens/AST/semantics/backend/runtime/docs/tests all align for the task's surface area.
- under the stricter current policy, `Done (Locked)` also means the shipped surface is hardened enough for the Java-close F1/self-sustained-hosting target, not just an earlier staged baseline
- Any Java-closeness claim must be backed by explicit support matrices and real fixture evidence.
- Validation is not deferred to a later phase; parser/semantic/backend/runtime/package/benchmark coverage lands as features are added.
- Any syntax or stdlib surface intentionally deferred beyond F1 must be documented as such; no ambiguity at closure.
- F1 stdlib expansion is for app/runtime language completion and self-sustained-hosting readiness, not yet a full external native library authoring/distribution ecosystem.
- `Class` reflection in F1 started as reflection-lite only, but the bar is now
  higher for `F1-97`: full Java-style reflection/invocation, annotation
  processing, metadata creation, and the migration/provider interface must land
  before the compiler/runtime rewrite proceeds.
- `awt` / `swing` / desktop UI work is explicitly out of scope for F1 and is deferred until after the self-sustained-hosting transition.
- if a bootstrap implementation can live in Pulse stdlib instead of Rust, F1 should prefer the Pulse-owned version and leave Rust with only the compiler/runtime/backend pieces that genuinely cannot move yet
- if a Pulse-owned helper is too elevated for public `stdlib` but still should not remain Rust-owned, it should be tracked toward the later `authorlib` split instead of being normalized into end-user API by accident
- F1 should push features toward self-sustained completion by default when that work fits the current lane cleanly
- F1 should not force a disproportionate self-sustained ownership move early just to claim it; when the later work is real but not correctly sequenced yet, record it in [FB_CARRY_FORWARD_TRACKER.md](/G:/Programming/Rust/PulseCode/docs/FB_CARRY_FORWARD_TRACKER.md) and leave the nearer-term task honest about what remains
- tasks may return from `Done (Locked)` to `In Progress` when the end-to-end feature is real but the ownership model is still knowingly bootstrap and a later F1/F2/F-A pass must finish the Pulse-facing contract honestly
- when backend/runtime/codegen work is required during an active F1 task, do not make broad ownership or lowering changes first and hope the suite holds; add an exact reproducer first, make the narrowest change that fixes that reproducer, and only then widen validation
- do not normalize stdlib workarounds around backend bugs as if they were final design; if Java-like source should work, the backend/runtime path should be fixed and the workaround removed
